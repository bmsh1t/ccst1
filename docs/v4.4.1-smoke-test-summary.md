## Probe Wiring Validation

**Direct hits:** json_inject_probe v4.4.1 fired correctly on the only JSON-body endpoint reachable in either lab (Juice Shop /rest/user/login):

- ✅ Signal A (JWT + admin role markers) → `sqli_auth_bypass` payload `' OR 1=1--` matched
- ✅ Signal B (SQL error fingerprint regex) → `'` payload triggered SequelizeDatabaseError pattern
- ✅ DEFAULT_LOGIN_SEEDS auto-discovery → `/rest/user/login` discovered from path patterns without manual seed
- ✅ Evidence file auto-generated: `findings/127.0.0.1:3000/poc/json_inject/sqli_auth_bypass_rest_user_login_email.json`
- ✅ AUTH_BYPASS_CLASSES generalization → same Signal A handler used for SQLi auth bypass (would also fire for nosql_op_injection / nosql_regex_bypass when target uses Mongo)

**Did not fire (and correctly so):**

- nosql_op_injection / nosql_regex_bypass → Juice Shop uses Sequelize+SQLite, not Mongo. No Mongo operators in baseline responses ⇒ correct silence.
- graphql_introspection → /graphql at Juice Shop returns Angular SPA HTML, not a GraphQL server. Baseline-clean requirement correctly suppresses false positive.
- ssti / open_redirect / path_traversal / xss (JSON variant) → no parameter on the probed login endpoint accepts unstructured strings that flow into a template / redirect / file path / HTML sink ⇒ correct silence.

**Scope clarification (not a gap):**

- DVWA is entirely form-data (`application/x-www-form-urlencoded` + `multipart/form-data`). json_inject_probe v4.4.1 is JSON-only **by design** — it is one fuzzing primitive within a larger toolchain. The supply-side layers (`recon_engine.sh` + `playwright-cli` browser probe + nuclei custom templates) are body-format-agnostic and **do** cover form-data. Validated 2026-05-17 by live capture of DVWA cmd_injection POST via `playwright-cli fill/click` — full content-type, request body, and response (uid=33 www-data) preserved. See `docs/v4.4.1-supply-side-coverage.md` and `evidence/127.0.0.1_8080/browser/supply-walk/exec_post_capture.txt`. → Earlier draft framed this as a "DVWA blind spot" — that framing was wrong; corrected.

## Pre-Existing Bug Fixed Mid-Smoke

While attempting to point autopilot at `127.0.0.1:3000`, hit:

```
ValueError: invalid IP/CIDR target  (target_paths.classify_target)
```

**Root cause:** strict-digits regex `r"[0-9./:]+"` matched but `ip_network()`/`ip_address()` both rejected the `host:port` form.

**Fix:** Added host:port detection block in `tools/target_paths.py` ahead of the strict-digits raise. Now handles:

| Input | Kind | Storage key |
|---|---|---|
| 127.0.0.1:3000 | ip | `127.0.0.1:3000` |
| 127.0.0.1:8080 | ip | `127.0.0.1:8080` |
| localhost:3000 | domain | `localhost:3000` |
| example.com:8443 | domain | `example.com:8443` |

Existing classifications unchanged for plain IPv4, CIDR, and bare domains. Full test suite (1476 tests) still passes.
