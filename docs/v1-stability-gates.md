# v1.0 稳定性验收门槛

本文档用于判断项目何时可以从当前稳定基线升级到 `v1.0.0`。

核心原则：不要靠感觉判断成熟度，而要证明系统可以反复跑通、可恢复、可复盘、可增强，并且红线规则始终生效。

## 1. 自动化流程稳定

从空上下文开始，至少完整跑通一次端到端流程：

```text
/target -> /scope -> /surface -> /autopilot -> /check-coverage
-> /validate -> /report -> /retrospect -> /memory-gc
```

验收标准：

- 每一步都能产出下一步可消费的上下文。
- 中断后可以通过 `/pickup` 或 context pack 恢复。
- autopilot 不会绕过红线规则。
- 没有发现漏洞时，会说明已覆盖范围、未覆盖范围和下一步建议，而不是直接结束。
- report 只输出有证据支撑的问题。

## 2. 记忆体系稳定

四层记忆必须职责清晰：

```text
skills 层：知道怎么做。
目标层：记住当前目标、范围、假设和进度。
知识库：提供思路发散和经验复用。
检查层：执行红线、覆盖率和完成门槛。
```

验收标准：

- 换上下文后仍能恢复当前目标状态。
- 红线规则始终生效，不会因上下文压缩或长任务而丢失。
- 目标记忆不会污染到另一个目标。
- 知识库只提供启发和检查角度，不替代证据。
- 记忆不会无限膨胀，能够通过 gc 或归档策略控制规模。

## 3. skills 增强闭环稳定

系统必须形成可持续增强闭环：

```text
实战/测试 -> /retrospect 复盘 -> 建议沉淀到 skill 或知识卡
-> 下次任务自动调用 -> 产生更好结果
```

验收标准：

- `/retrospect` 能指出经验应该进入哪个 skill、知识卡或规则文件。
- 新增知识不会让模型只盯单一方向测试。
- skill 更像方法论和工作流，不写成僵硬死规则。
- 至少经过 3 次不同任务验证后，系统表现确实变好：重复更少、恢复更快、覆盖更完整。

## 4. v1.0 前硬门槛

打 `v1.0.0` 前必须满足以下条件：

```text
关键测试全过
runtime doctor 无漂移
命令无冲突
敏感文件不进 git
红线测试通过
autopilot normal/deep 都能稳定跑
上下文中断恢复成功
至少完成 3 个授权或实验目标的端到端流程
```

推荐版本节奏：

```text
v0.1.0 = 当前稳定基线
v0.2.x = 使用中修复 automation 和 memory 问题
v0.5.0 = 复盘增强闭环稳定
v1.0.0 = 连续多轮实战或实验验证后再打
```

## 结论

`v1.0.0` 不代表功能数量足够，而代表系统已经证明自己可以稳定执行、稳定恢复、稳定复盘、稳定增强，并且红线规则不会丢失。
