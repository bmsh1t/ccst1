## P1 — Optimization items that block expert-grade autonomy

### P1.1 Auto-invoke `sibling_generator.py` after every validated primary finding

**Observed:** Juice Shop h1 validated `/rest/admin/application-configuration`. The autopilot did not auto-enumerate `/rest/admin/application-version`, `/rest/admin/`, `/rest/admin/*` until an operator constructed the curl loop manually. Two of the three siblings produced new evidence (Express stack trace on `/rest/admin/`; version disclosure on `application-version`).

**Required change:**
- `agent.py` `validate_finding` hook → on success, push sibling probe seeds to a `worker_queue.jsonl`
- Autopilot loop drains queue between hypotheses
- Sibling generator reads endpoint shape (`/rest/admin/X`) and produces 5–10 morph variants (HTTP method swap, path-segment swap, version swap, ID swap, suffix probe)

**Effort:** Medium (2–4 h). The tool exists; the auto-invocation glue does not.

### P1.2 Bypass-template auto-chain after a 403/404 signal

**Observed:** Juice Shop `/ftp/coupons_2013.md.bak` and `/ftp/package.json.bak` returned 403. The autopilot did not auto-attempt the well-documented Juice Shop poison-null-byte bypass (`%2500.md`). Manual replay extracted both files (4263 B legacy `package.json.bak`, 131 B coupon codes).

**Required change:**
- `agent.py` recon-result handler → on 403/404 against an exposure-path candidate, queue 5 canonical bypass payloads:
  1. Trailing null-byte (`%2500.{ext}`)
  2. Path normalization (`//`, `/./`, `/../{file}`)
  3. Header bypass (X-Original-URL, X-Rewrite-URL)
  4. Method swap (POST/PATCH/HEAD)
  5. IIS short-name (when stack matches)
- `tools/bypass_403.sh` already implements (1)–(4); wire it to the autopilot finish queue

**Effort:** Low (1–2 h, mostly orchestration).

### P1.3 Auth-aware hunt loop — DVWA-style CSRF token rotation

**Observed:** DVWA login requires a per-request `user_token` regenerated on every `/login.php` GET. `/autopilot 127.0.0.1:8080 --auth-file` cannot construct this profile automatically — the autopilot would need to:
1. GET `/login.php` → parse `user_token` from HTML
2. POST `/login.php` with `username, password, user_token, Login` → capture Set-Cookie
3. For every subsequent action that has a form-token, re-fetch the page → extract token → submit

**Required change:**
- `tools/auth_session.py` (new) — token-aware login engine
  - Inputs: target, login URL, login_field map, csrf_field_selector (regex), success_indicator (302 to /index.php, or absence of "Login failed")
  - Outputs: refreshed cookie jar + helper that, given a request, pre-fetches the page and stitches the rotated token in
- Wire into `tools/hunt.py --auth-file` so a JSON profile can declare the rotation contract:
  ```json
  {"login":{"url":"/login.php","method":"POST","fields":{"username":"admin","password":"password"},"csrf":{"selector":"name=['\"]user_token['\"]\\s+value=['\"]([^'\"]+)"},"success":"302"}}
  ```

**Effort:** High (1 day). This is the single biggest unlock for stateful-app coverage.

### P1.4 Auto-rotate to a different lane when `next_question` stalls

**Observed (Juice Shop h6):** All 4 open-redirect bypass payloads returned 406. The autopilot would normally need an explicit kill-condition + lane-rotation directive. Operator did this manually (killed h6, moved to h4/h5). The autopilot loop does not yet auto-rotate based on "N consecutive identical-status responses on the same hypothesis."

**Required change:**
- `agent.py` working_hypothesis tracker → add `consecutive_uninformative_attempts` counter
- When counter ≥ 3, log `[ROTATE]` audit line and call `pick_next_lane` (which already exists)
- Lane rotation order: web2 → web2 with different transport → JS-read intel → source-intel → kill

**Effort:** Low (1 h glue + audit line).

## P2 — Quality-of-life improvements that compound

### P2.1 Capture `working_hypothesis` transitions in `hunt-memory/journal.jsonl` automatically

**Observed:** This entire run's hypothesis trace (`h1`...`h9` for DVWA, `h1`...`h6` for Juice Shop) had to be reconstructed manually for the attack_paths.md. The autopilot loop tracks `working_hypothesis` internally per `agent.py` but does not flush hypothesis-state transitions to `journal.jsonl` for later reporting.

**Required change:** `agent.py` → emit a `{"event":"hypothesis_transition","from":"h2","to":"h3","reason":"validated; pivot to siblings"}` row on each state change.

**Effort:** Low (30 min).

### P2.2 Recon-engine `Recon: Skipped` false-negative regression test

**Observed:** v4.4.2 fixed 5 sequential bash defects that caused `Recon: Done` to mis-report `Recon: Skipped` in hunt.py's dashboard. This run confirms the fix held (Juice Shop `Recon: Done | Scan: Done` after `recon_engine.sh` exit code 0). But there's no e2e test that pins the dashboard output.

**Required change:** Add `tests/test_hunt_dashboard.py` that:
1. Stubs `subprocess.Popen` to return exit code 0 from recon_engine
2. Invokes `hunt.py --target 127.0.0.1:3000 --quick`
3. Asserts dashboard contains `Recon: Done` (not `Skipped`)

**Effort:** Low (30 min). Pins v4.4.2 regression.

### P2.3 Verbose-error fingerprint extraction

**Observed (Juice Shop AP-3):** `GET /rest/admin/` → 500 with full Express stack trace exposing build paths (`/juice-shop/build/routes/angular.js:42:18`), framework version (`Express ^4.22.1`), and routing structure. The autopilot scanner did not flag this as `info_disclosure` because the `error_template` lane only matches a small whitelist of error strings.

**Required change:** Add stack-trace regex to `findings/<target>/exposure` lane:
- `at [^\s]+\.js:\d+:\d+` → Node/Express stack
- `at [^\s]+\.py", line \d+` → Python stack
- `^\s+at .+\(.+:\d+\)` → Java/.NET stack
- `Microsoft\.AspNetCore\.` → ASP.NET

**Effort:** Low (1 h).

### P2.4 KeePass / archive-format magic-byte detection in exposure scanner

**Observed (Juice Shop AP-2):** `/ftp/incident-support.kdbx` is a KeePass 2.x database (magic `03 d9 a2 9a 67 fb 4b b5`). Exposure scanner returned 200 but did not flag it as sensitive. A KeePass DB exfil is critical (offline cracking → potential infrastructure credential pivot).

**Required change:** Exposure scanner → after fetching candidate, check first 8 bytes against magic-byte table:
- `03 d9 a2 9a 67 fb 4b b5` → KeePass 2.x (CRITICAL)
- `7B 4B 65 79 50 61 73` → KeePass XML
- `50 4B 03 04` → ZIP (potentially backup archive)
- `1F 8B 08` → gzip (potentially backup tar.gz)
- `28 B5 2F FD` → zstd
- `42 5A 68` → bzip2

**Effort:** Low (1 h).

## P3 — Nice-to-have items deferred until P1/P2 land

### P3.1 DVWA security-level pivot enumeration

`security=low` is the canonical bug surface. `security=medium` and `security=high` are exercises in bypass discovery. The autopilot does not currently flip the cookie to test each level.

**Required change:** Optional `--security-levels low,medium,high` flag for DVWA-shaped targets.

**Effort:** Low (1 h). Only useful for DVWA-like training labs.

### P3.2 Web3 / JSON-RPC surface follow-up

**Observed (Juice Shop):** 64 JS-extracted endpoints included `mainnet.infura.io/v3`, `eth-mainnet.alchemyapi.io/v2`, `arb1.arbitrum.io/rpc`, etc. The autopilot did not auto-construct JSON-RPC SSRF / parameter-injection probes against these.

**Required change:** New `tools/web3_rpc_probe.py` — for any `*.infura.io/v3/<key>` style endpoint, attempt unauth JSON-RPC `eth_blockNumber`, `net_version`, `eth_getBalance` to detect:
- Anon RPC access (no API key required)
- Key disclosure (key embedded in JS bundle)
- SSRF pivot via the gateway

**Effort:** Medium (3 h). Only fires when JS-extracted endpoints match Web3 patterns.

### P3.3 rtk wrapper compatibility — bash for-loops

**Observed (capability gap #1 from session):** `for path in a b c; do curl "http://.../$path"; done` runs the first curl, then iterations 2+ fail with "command not found: curl". The rtk wrapper appears to intercept the loop iteration body and lose the curl binding.

**Workaround used:** Unroll loop into individual curl lines.

**Required change:** Either:
- (a) `rtk` should not intercept curl inside `for` loops
- (b) Document the workaround in `CLAUDE.md` so autopilot LLM avoids the pattern
- (c) Provide `tools/batch_curl.py` wrapper that handles the loop in Python

**Effort:** Investigation needed (out of scope for this repo if rtk is global). Documenting (b) is the cheapest fix.

### P3.4 `coverage_matrix find-gaps` integration into post-finish summary

**Observed:** The autopilot `finish` gate already self-enforces F3 (coverage matrix find-gaps must be empty). But the **dashboard summary** at end of run does not show which cells are tested vs n_a vs gap. An operator finishing a run cannot tell at a glance whether coverage is exhaustive without running `find-gaps` manually.

**Required change:** `hunt.py` dashboard → after `Recon: Done | Scan: Done`, append:
```
  Coverage: 47 cells tested_clean, 3 tested_finding, 12 n_a, 0 gap
```

**Effort:** Low (1 h).

## Recommended Sequencing for v4.5.0

If the goal is "true single-command autopilot E2E on auth-gated targets," ship in this order:

1. **P2.2** (regression pin, 30 min) — prevent v4.4.2 fixes from regressing
2. **P1.4** (lane auto-rotate, 1 h) — biggest UX win per hour invested
3. **P1.1** (sibling auto-enumeration, 2–4 h) — biggest finding-per-hour win
4. **P1.2** (bypass-template chain, 1–2 h) — depends on P1.1 plumbing
5. **P1.3** (auth-aware loop, 1 day) — gates DVWA-shape coverage completely
6. **P2.1, P2.3, P2.4, P3.4** in a single batch (4 h) — observability + sensitive-file detection
7. **P3.1, P3.2** as time permits

Estimated total: ~3 dev-days for P1+P2; another 1 day for P3. After P1.3 lands, `/autopilot <target> --normal --auth-file dvwa.json` becomes a single-command run.
