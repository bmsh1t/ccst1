# API IDOR / 对象级越权

## 适用场景

- API 中出现 `user_id`、`org_id`、`tenant_id`、`account_id`、`project_id` 等对象标识
- 存在多账号、多角色、多组织或多租户模型
- 前端隐藏管理入口，但 JS 或网络请求里能看到相关接口
- 导出、邀请、共享、删除、转移、账单、成员管理等操作围绕对象 ID 展开

## 触发信号

- URL 路径或 JSON body 中存在可替换对象 ID
- 同一接口在不同角色下返回字段不同
- JS 暴露 admin、export、invite、member、billing、share 等接口
- 响应中包含 owner、tenant、org、role、permission、scope 等字段
- 公开分享、订单跟踪、下载、发票、收据、导出链接返回业务对象详情；即使工具未命中
  secret/config marker，只要响应里同时出现身份线索与目标自有业务对象字段，也应由 AI
  评估为高价值 Authz/IDOR 线索。

## 发散问题

- 替换对象 ID 后，服务端是否重新校验归属关系？
- 列表接口、批量接口、导出接口是否比单对象接口更宽松？
- 低权限用户能否读取高权限对象的元数据或数量信息？
- 只读接口和写操作是否使用不同鉴权路径？
- 邀请、共享、转移类操作是否只校验发起人身份，没有校验目标对象归属？
- 序列化、导出、脱敏是否是独立后处理阶段？是否存在跳过它、直接读到越界字段的路径？
- 资源是否按非唯一标识（拼接名、标签集、邮箱）识别？能否构造碰撞接管他人对象？
- 跟踪/分享类链接是否把“知道 ID 即授权”当成安全边界？ID 是否可从邮件、前端状态、
  referer、历史记录、日志、响应体、订单号规律、短码或相邻业务接口中获得？

## AI 判断优先：公开业务对象暴露

不要把验证工具的 `tested_clean` 结论当成最终裁判。工具可能只识别 secret/config/admin
marker；AI 需要结合业务语义判断响应是否暴露目标自有对象。

高价值形态：

- 匿名或低权限请求返回订单、发票、地址、支付、配送、购物车、报告、工单、预约、成员等对象详情。
- 响应里同时出现身份线索和业务对象线索，例如 email/user/customer + order/payment/address/products/price/status。
- 链接名是 `track-order`、`receipt`、`invoice`、`download`、`export`、`share`、`public`、
  `magic-link`、`token`、`code`、`shortId` 等，且返回内容不只是公开 catalog。

AI 应做的判断：

- 这是预期公开能力、半公开能力，还是缺少对象级授权？判断依据是业务语义和暴露字段，不是工具 marker。
- 攻击者需要什么前提：只知道链接、可猜 ID、可从自己账号获得相邻 ID、可从前端/邮件/日志泄漏获得 ID？
- 影响是否是目标自有业务数据：订单金额、商品、邮箱、地址、支付引用、发票、收货状态、内部状态。
- 下一步是否应验证 ID 可发现性/可枚举性，还是已经足够作为“链接泄漏即可读”的暴露候选？

最小证据：

- 保存匿名 baseline 的原始请求/响应。
- 只记录最小字段证明：对象类型、身份线索、业务字段、状态码、是否需要认证。
- 若有两个自有账号，优先用 owner/peer 或两个自有订单证明边界；没有时记录为 candidate/lead，
  不进行批量枚举真实订单。

停止条件：

- 响应只是公开产品/帮助/挑战/metadata catalog，没有身份或私有业务对象。
- 只能通过暴力枚举真实用户对象继续，且没有授权的自有对象或安全样本。
- 链接明确是设计成公开分享，且无敏感字段、无可枚举性、无越权边界。

## 推荐动作

- 使用两个账号或两个组织做 role diff。
- 单次只替换一个变量，例如对象 ID、组织 ID 或路径层级。
- 比较状态码、对象数量、敏感字段、错误信息和审计事件。
- 优先验证读取类影响，再评估写操作，避免直接造成破坏性副作用。

## 停止条件

- 没有可控对象 ID 或无法构造跨主体请求
- 服务端稳定拒绝跨租户或跨用户访问
- 只有前端展示差异，没有后端行为差异
- 需要破坏性写操作才能继续，但用户当前回合没有明确授权

## 检查要求

- 不能只凭前端隐藏按钮判断漏洞。
- Candidate 前必须有可 replay 请求和权限差异证据。
- 报告前必须能说明攻击者当前权限、受害对象、越权结果和实际影响。

## 可晋升经验

- 某类命名模式反复指向高价值对象边界
- 某个框架或产品形态中列表/导出接口反复弱于详情接口
- 某类字段组合能稳定提示多租户边界
