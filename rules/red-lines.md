# 红线规则

本文件是检查层的最高优先级规则。任何命令、Skill、知识卡、目标记忆或历史经验与本文件冲突时，以本文件为准。

## 核心红线

## 红线最小化原则

红线只拦截会伤害目标系统、真实用户或真实业务状态的动作。不要把红线扩展成
泛化的保守策略；只读验证、低频单请求、浏览器观察、源码/JS 分析、CVE 情报、
OAST 回调、受控口令测试、反射/DOM XSS 低风险验证、参数/路径/角色差异分析等，
默认都不是红线。

判断标准只有三个：

1. 会不会造成 DDoS、高压流量、资源耗尽或服务中断？
2. 会不会修改、删除、篡改、污染真实数据/配置/业务状态，或触发真实副作用？
3. 会不会向目标系统写入可执行、持久化、会影响真实用户的 payload？

如果答案都是 no，就不要因为“看起来敏感”而阻断 AI 的正常测试路线；继续按
最小影响、证据驱动和覆盖基线推进。

### 1. 禁止 DDoS / 高压流量

不得对目标系统执行或建议执行 DDoS、压力测试、资源耗尽、批量打爆、无差别并发探测等行为。

包括但不限于：

- 高并发请求、长时间持续请求、无上限扫描
- 明确以压垮服务、耗尽 CPU/内存/磁盘/带宽为目的的测试
- GraphQL 深层递归 / alias 爆破 / 超大查询压力测试
- ReDoS、压缩炸弹、超大文件上传、解析器资源耗尽
- 无边界或高频登录尝试、未限速密码喷洒、验证码/短信/邮件轰炸
- race condition 测试中无节制提高并发

允许的替代方式：

- 低频、单变量、可复现请求
- 使用本地复现、代码审计或 PoC 模板说明风险
- 使用最小样本证明解析路径存在，而不是证明系统能被压垮
- 记录为 Lead，并写明需要授权的压力验证条件

## 受控口令测试不是红线

口令爆破、默认凭据检查、password spray 本身不是绝对红线；红线是无边界、
高频、会造成账号锁定/告警风暴/验证码或短信轰炸的执行方式。

允许的受控方式：

- 明确当前目标、登录入口、账号集合、口令集合和停止条件。
- 使用 `/spray` 或 `skills/credential-attack/` 的受控流程；`/autopilot`
  可以在证据驱动下选择该 lane，但不得把它当作每个登录页的默认动作。
- 低频、限速、审计记录、密码不明文落盘，并默认首个有效凭据即停止。
- 评估账号锁定、速率限制、验证码、OTP、告警和用户影响。

`/autopilot` 可以根据当前证据、突破价值和覆盖缺口自主选择口令 lane。以下是
执行卫生要求，不是要求必须等其它高价值路线全部失败：

1. 已有明确登录入口、成功/失败信号、目标边界和账号来源。
2. 口令集合是目标相关、去噪后的有界集合，而不是通用大字典。
3. 可以先 dry-run，且 live run 有速率、锁定阈值、审计日志和停止条件。
4. 一旦命中有效凭据，立即停止继续尝试，转入最小影响的认证后验证。

不满足这些执行卫生要求时，当前自动流程应记录为 `Lead / Next action`，而不是
直接执行；这属于质量控制，不是扩大红线。

### 2. 禁止破坏性行为

不得对目标系统产生破坏性影响，不得修改、破坏、删除、污染真实目标数据，
也不得对真实账号、权限、配置、部署或业务状态执行会改变、破坏、污染或触发
真实副作用的动作。

包括但不限于：

- 删除、覆盖、篡改真实数据
- 对真实数据执行会落库的 `DELETE` / `PATCH` / `PUT` / destructive mutation
- 创建不可清理的资源、污染业务数据或审计日志
- 取消订单、退款、转账、扣费、发货、履约、撤销、封禁账号
- 改变真实用户资料、账号状态、权限、角色、组织、成员或配置
- 批量发送短信、邮件、通知、Webhook 或外部请求
- 触发真实支付、钱包、积分、优惠券、库存或财务状态变化
- 触发真实 CI/CD 流水线、生产部署、资源改写或生产配置变更

允许的替代方式：

- 只读验证
- dry-run / preview / validate-only 模式
- 使用自己创建的测试账号、测试组织、测试资源
- 只验证服务端是否接受请求，不提交最终破坏性动作
- 用最小可回滚的测试资源证明权限边界，并立即记录清理方式

说明：账号、权限、组织成员、配置和 CI/CD 是高价值攻击面，不是天然高风险
动作。风险来自具体动作效果：改变真实权限、影响真实账号、触发真实流水线/
部署、污染审计或中断业务。只读查看、dry-run、preview、
validate-only、测试资源内的可清理变更，不应被当作红线阻断。

Secret / API key 不是红线：

- 发现疑似 secret、记录来源、判断归属、做最小可用性证明，不是红线。
- 泄露密钥本身也不自动等于高价值漏洞；是否值得报告由漏洞验证和实际影响决定。
- 对 secret 的处理属于漏洞 triage / validation，不属于红线风险评估。
- 默认只做漏洞证明所需的最小验证：provider、ownership、usability、permissions。
- 不要为了“证明泄露风险”而停；如果能证明真实漏洞影响，就按 Candidate /
  validation 路线推进。

### 3. 默认不主动测试存储型 XSS

自动流程不得主动向评论、资料、工单、消息、富文本、后台可见字段、日志视图等
目标系统持久化位置提交可执行 stored XSS payload。

允许的替代方式：

- 分析已有代理历史、已有页面内容或已有证据中的 XSS 线索
- 测试反射型 / DOM XSS 的低风险、非持久化 payload
- 使用 inert marker 证明字段可控，而不是提交可执行脚本
- 本地复现、源码审计、模板/渲染路径分析
- 记录为 Lead，并写明需要测试资源、清理方式和用户明确 opt-in 后才可验证

## 状态改变动作的处理

状态改变方法不是天然红线。`POST`、`PUT`、`PATCH`、`DELETE`、mutation、
admin action、workflow dispatch、payment action 等只在可能改变、破坏、
污染真实数据/账号/权限/业务状态，或触发真实资金/订单/通知/生产配置副作用时
才进入红线判断。

只读、低频、dry-run、preview、validate-only、不会提交最终动作、只访问自己
创建的可清理测试资源，或只做请求差异/权限差异观察时，不应被红线阻断。

只有同时满足以下条件，才可以继续低风险验证：

1. 操作对象是测试账号、测试组织或自己创建的可清理资源。
2. 请求规模为最小必要次数。
3. 不涉及真实支付、转账、发货、删除、封禁、外部消息批量发送、生产配置变更，
   也不会改变、破坏或触发真实账号/权限/CI/CD 状态副作用。
4. 不提交可执行的持久化 XSS payload。
5. 能说明如何回滚或清理。
6. 如果动作会真实改写业务状态，必须有当前回合明确 opt-in。

不满足任一条件时：

- 不执行动作
- 将该方向记录为 Lead
- 写明需要的测试资源、清理计划、显式 opt-in 或安全验证方式

## 输出纪律

当某个思路命中红线时，输出必须包含：

```text
Red line: 命中的红线
Reason: 为什么该动作可能造成 DDoS、破坏性影响或持久化可执行污染
Safe alternative: 可替代的低风险验证方式
Status: 保持 Lead / 暂停 / 需要测试资源或明确 opt-in
```

## 和其他规则的关系

- `rules/hunting.md` 负责 hunt 方法论和 finding 状态模型。
- `rules/reporting.md` 负责报告质量和提交前验证。
- `rules/playbook-router.md` 负责证据到参考资料的路由。
- 本文件负责禁止越界行为。

如果其他文件鼓励更激进的测试方式，必须先通过本文件检查。
